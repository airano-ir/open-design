背景

每個模型他都試。他要的是同一塊創作介面。

Stuart 經營一家一人顧問公司 Connectio,幫企業把 AI 驅動的 app 做出來 — 同時也經營 Let’s Build,一個講怎麼用 AI 做東西的 YouTube 頻道。(在那之前,他是 FlutterFlow 的成長駭客。)跟上最新進度就是他的工作,所以他不斷在各種新工具、新模型和 coding agent 之間切換。他需要的不是某一個完美的模型 — 而是一塊底下模型怎麼換、它都穩穩不動的創作介面。

他原本用的是 Claude Design,也認同它的方向。但它把他綁死在同一家公司的模型上,而對一個每週都在換模型的人來說,這種綁定顯然不合適。後來,一位同行發文提到一個開源的替代方案 — “我一向盡量支持開源社群,所以就試了一下。”

Claude Design 能做的它全都能做 — 但用的是我自己選的模型。而且它還更進一步,因為它沒被綁手綁腳。
Stuart Gardoll · Connectio
實證 · 一條 app 工作流

一款已上架的 app,在 Open Design 裡成形。

Stuart 的 Read It — 一款能把任何文章變成語音的 app — 已經上架 App Store 和 Google Play。介面就是在 Open Design 裡成形的:閱讀動線、載入的節奏編排,還有那些讓它感覺有溫度的細小時間點。他在裡面把方向設計好,再把產物交給他的 coding agent,在自己的 Flutter 技術棧裡把它做出來。

“我需要什麼就做什麼,然後交給我慣用的 CLI。整個流程順得很。”

實證 · 一條內容工作流

他影片裡的動態圖像 — 都在 Open Design 裡做的。

另一條工作流是媒體。Stuart 有些 Let’s Build 影片,動態圖像就是用 Hyperframes 在 Open Design 裡做出來的:他描述自己要什麼、套上自己的品牌配色,Open Design 就把它生出來、即時預覽,再匯出成 MP4 — 直接就能丟進剪輯裡,而且是他觀眾期待的那種成品級品質。

“我最新那支影片,所有動態圖像都是用 Open Design 做的。”

而且還不只這些

不管他在做什麼,都是同一塊介面。

從一個 repo 做出原型

他會拿一個現成的 GitHub repo,在 Open Design 裡重新想像一遍 — 把一個粗糙的工具,變成乾淨、真的能出貨的東西。

能跑的產物

一個早上的實驗,把 Hacker News 變成一份 Flipboard 風格的報紙 — “成果驚人”,接著他就直接把它放進軟體裡。

他自己的設計系統

不是用預設值 — 他拿自己的設計系統、可重複用的視覺方向來實驗,再照著它們生成。

為什麼它成了他的預設

它能接進他手上已經在用的一切。

Open Design 剛好嵌進 Stuart 技術棧的其餘部分:他從 Figma 匯入,乾淨俐落地交棒給他的 CLI 和 coding agent,還很想把它接進日常在用的專案工具裡。“我很喜歡這個平台跟其他一切都連得這麼緊。”對一個什麼工具都要試一輪的人來說,這就是他不再到處比較的原因。

結論
“我根本沒需要再去別的地方 — 它功能齊全得驚人。”
